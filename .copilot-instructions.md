# Web Serial Monitor - Copilot Instructions

This document defines custom agent behaviors and skills for working with the Web Serial Monitor project.

## Agent: Web Serial Monitor Development

**Purpose:** Assist with development, debugging, and feature additions for the lightweight serial port monitor.

**Scope:** Vanilla JavaScript, Web Serial API, DOM manipulation, and pure CSS.

### Core Capabilities

#### Code Understanding
- Analyze the modular class structure (SerialPortManager, HexViewer, UIManager)
- Explain Web Serial API interactions
- Review DOM event binding and state management
- Debug JavaScript execution flow

#### Development Tasks
- Add new features while maintaining simplicity
- Fix bugs in serial communication
- Optimize performance without dependencies
- Improve accessibility and responsiveness
- Write unit tests (vanilla JS preferred, no test frameworks)

#### Documentation
- Update README.md with new features
- Maintain architecture documentation
- Add inline code comments for complex logic
- Keep requirements.md current

### Preferred Code Style

- **No frameworks** — Vanilla JavaScript only
- **Modular classes** — Keep related functionality in classes
- **Event-driven** — Use event listeners for UI interactions
- **Comments** — JSDoc-style comments for complex functions
- **CSS Grid/Flex** — Modern CSS layout, no frameworks
- **Dark theme** — Maintain consistency with existing UI

### Common Tasks

#### Adding a New Feature
1. Identify which class needs modification (SerialPortManager, HexViewer, or UIManager)
2. Add HTML elements to public/index.html
3. Add CSS rules to public/styles.css
4. Implement JavaScript logic in app.js
5. Test in supported browsers
6. Update requirements.md if needed
7. Update README.md with feature documentation

#### Debugging Serial Communication
1. Check browser console for errors
2. Verify Web Serial API availability
3. Review SerialPortManager connection state
4. Check for incomplete reads (look at reader.read() loop)
5. Verify escape sequence processing in write()

#### Optimizing Performance
- Keep classes focused and single-responsibility
- Avoid unnecessary DOM updates (batch updates together)
- Use efficient event delegation where applicable
- Profile with Chrome DevTools (especially for large data transfers)

### File Structure to Maintain

```
public/
  index.html        # Single HTML file, semantic structure
  styles.css        # Organized by component, dark theme
  app.js           # Three main classes, ~300-400 lines
requirements.md     # Clear browser and OS requirements
README.md          # User-focused documentation
package.json       # Minimal, no runtime dependencies
```

## Skills

### skill:serial-api
**Description:** Understanding and implementing Web Serial API features  
**Trigger:** Questions about serial port communication, connection issues, data transmission  
**Context:** Review SerialPortManager class, understand async/await patterns for serial I/O

### skill:hex-viewer
**Description:** Hexadecimal data formatting and line calculations  
**Trigger:** Hex display issues, byte formatting, alignment problems  
**Context:** Review HexViewer class, understand padding and line-wrapping logic

### skill:ui-state-management
**Description:** DOM state synchronization and event binding  
**Trigger:** UI not updating, button state issues, form interactions  
**Context:** Review UIManager class, check event listener setup, verify state flags

### skill:accessibility-improvements
**Description:** Making the app more accessible (ARIA, keyboard nav, screen readers)  
**Trigger:** Accessibility issues, keyboard shortcut requests, ARIA label improvements  
**Context:** Review HTML for semantic structure, suggest ARIA enhancements

### skill:performance-tuning
**Description:** Optimizing rendering and serial data handling for high-speed connections  
**Trigger:** Lag with high baud rates, memory issues, UI freezing  
**Context:** Profile with DevTools, consider debouncing updates, batch DOM changes

## Anti-Patterns to Avoid

❌ **Do Not:**
- Add npm dependencies (defeats the zero-dependency goal)
- Use frameworks (React, Vue, Svelte, etc.)
- Create build processes or bundlers
- Use minification or transpilation
- Add testing frameworks (use vanilla test functions if needed)
- Over-engineer solutions
- Break classes into too many small files

✅ **Do Instead:**
- Keep code simple and readable
- Use native browser APIs
- Maintain the 3-class architecture
- Write tests as simple functions in separate files if needed
- Prefer CSS over JavaScript for styling
- Keep file count minimal

## Testing Approach

Since we avoid testing frameworks:

1. **Manual testing** — Test in Chrome, Edge, and Opera
2. **Browser DevTools** — Use console and debugger
3. **Test devices** — Have real serial hardware for testing
4. **Simple test files** — Create `test-xxx.js` files with functions that log results
5. **Example:** Create `public/test-hex-viewer.js` to test HexViewer class in isolation

## Browser Compatibility Checklist

When adding features, verify:
- [ ] Works in Chrome 89+
- [ ] Works in Edge 89+
- [ ] Works in Opera 76+
- [ ] Graceful degradation for unsupported APIs
- [ ] Responsive on mobile (if applicable)
- [ ] Works offline (after initial load)
- [ ] No console errors in production

## Documentation Requirements

- Every new feature needs README.md update
- Update requirements.md if new browser or OS requirements added
- Add inline comments for complex algorithms (like hex formatting)
- Maintain JSDoc comments on class methods
- Keep this instructions file updated with new patterns or anti-patterns
